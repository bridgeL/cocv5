# 开发工作流程

本文档描述项目的标准开发流程，确保开发过程规范、可控、可追溯。
你是项目开发者。

## 1. 文档编写

### 输出物
1. `docs/task/{编号}-{任务名}.md`
2. `docs/interface/{功能名}.md`
3. `docs/ws/{功能名}.md`
4. `docs/database/{功能名}.md`

### 步骤
1. 与用户沟通，充分理解需求背景和目标，拆解任务步骤，编写任务文档
2. 如果有新增接口，编写对应的接口文档，文档规范见后续的 `接口文档规范`
3. 如果有新增ws交互消息格式，在对应的功能的ws文档里编写该格式
4. 如果有新增数据表，在对应的功能的数据表文档里编写 数据表名、字段名、类型和对应含义；所有数据表设计时只允许有主键约束，不允许有任何外键
5. 提交给用户确认
6. 根据反馈反复修改，直到用户确认通过

### 接口文档规范

必须包含：
- 接口名
- 接口方法
- 接口地址
- 接口请求参数
- 接口正常返回
- 接口异常返回

接口方法只允许有GET / POST

- POST传递参数时必须是body json
- GET传递参数时必须是query param

接口返回值必须是字典，包含字段
- code（四位数字的字符串）
- msg（失败原因）
- data（返回的具体数据在data中）

code含义

- 0000代表默认正常返回
- 1111代表失败返回
- 可以有多个自定义值
    - 0开头的是正常返回
    - 1开头的是异常返回
    - 后续三位用于区分不同的返回

---

## 2. 开发代码

### 步骤
1. 根据任务文档逐个完成任务
2. 编写代码完成功能
3. 编写测试代码进行自测，修复遇到的bug
4. 如遇计划外问题，及时与用户沟通

---

## 3. 用户评价

### 步骤
1. 等待用户确认
2. 如果用户反馈有bug，你需要回到第三步重新开发并测试
3. 反复修改直到用户确认测试无误

---

## 4. 代码提交

### 步骤
1. 整理代码变更
2. 按照 [git.md](git.md) 规范编写提交信息
3. 执行提交
